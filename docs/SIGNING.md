# コード署名（自己署名証明書）

Snapperkun はアップデートで `.app` を入れ替えるため、**ビルドごとに署名が変わると
macOS のアクセシビリティ権限 (TCC) が無効化**されます（設定上は有効に見えるのに
ホットキーが効かず、権限を削除→再登録しないと動かない、という症状）。

これを防ぐには、**ビルドをまたいで同一の安定した署名アイデンティティ**で署名します。
ここでは無料の自己署名証明書を使う手順を示します（Gatekeeper の警告は残りますが、
アクセシビリティ権限はアップデート後も保持されます）。

## 1. 自己署名のコード署名証明書を作成

Keychain Access（キーチェーンアクセス）で:

1. メニュー **キーチェーンアクセス > 証明書アシスタント > 証明書を作成…**
2. 名前: `Snapperkun Self-Signed`（任意。これが署名アイデンティティ名になる）
3. 固有のタイプ: **コード署名**
4. 「デフォルトを無効にする」にチェックし、**有効期間を長め（例: 3650 日）** に設定
5. 作成してログインキーチェーンに保存

> 有効期間が切れると署名できなくなるので長めに。

## 2. ローカルでの署名（任意・開発用）

```sh
SIGN_IDENTITY="Snapperkun Self-Signed" bash Scripts/bundle.sh release
```

`SIGN_IDENTITY` 未指定なら従来どおり ad-hoc 署名になります。

## 3. CI（GitHub Actions）で同じ証明書を使う

リリースは CI が生成するため、**CI でも同一証明書で署名**する必要があります。

### 証明書を .p12 で書き出す

Keychain Access で作成した証明書（と秘密鍵）を選択 → 右クリック **書き出す…** →
`.p12` 形式でパスワードを付けて保存。

### base64 にして GitHub Secrets に登録

```sh
base64 -i Snapperkun.p12 | pbcopy   # クリップボードにコピー
```

リポジトリの **Settings > Secrets and variables > Actions** に以下を登録:

| Secret 名 | 値 |
|---|---|
| `SIGNING_CERTIFICATE_P12_BASE64` | 上記の base64 文字列 |
| `SIGNING_CERTIFICATE_PASSWORD` | .p12 のパスワード |
| `SIGNING_IDENTITY` | 署名アイデンティティ名（例: `Snapperkun Self-Signed`） |

これらが揃うと、リリースワークフローはこの証明書で署名します。未設定なら
ad-hoc 署名にフォールバックします（その場合は権限が保持されません）。

## 4. 移行時の一回だけの再許可

ad-hoc 版から自己署名版に切り替わる初回だけ、署名が変わるため再許可が必要です:

1. システム設定 > プライバシーとセキュリティ > アクセシビリティ で
   古い Snapperkun のエントリを削除
2. 自己署名版を起動して再度許可

以降、同じ証明書で署名されたアップデートでは権限が保持されます。
